# Personal OS — Architecture (Stage 1)This document explains how the system is structured so you can understand it, modify it, and grow it over time.---## The Big PictureThe system has five layers stacked on top of each other:```TELEGRAM (input)    ↓VERCEL (server + dashboard)    ↓CLAUDE API (intelligence)    ↓SUPABASE (memory)```GitHub sits next to Vercel as the code store.When you send a message to your Telegram bot, the message hits a Vercel serverless function (`pages/api/telegram.js`), which talks to Claude (to reason about the message), reads/writes Supabase (to remember), and replies via Telegram. The dashboard you open in a browser is also served by Vercel and reads directly from Supabase.---## Core Concept: AspectsThe system is built around **aspects** — the 12 named areas of your life. Each aspect has a `type`:- `doing` — where life is actively unfolding (work, body, money, relationships)- `inner` — where you locate yourself (reflections, beliefs, spiritual)- `derived` — the live read on how the others interact (Lifestyle)Each type has different surfaces. Doing aspects have Vision, Sub-aspects, Ideas, Jobs, Done. Inner aspects have Vision, Reflections, Ideas. Compass is special — it has sections for Beliefs, Purpose, Mission, Standpoints. Lifestyle is special — it has Default and Present states.This typing matters because it tells the system how to engage with each aspect. The Question Engine asks different kinds of questions about an inner aspect than a doing aspect. The Mapping Engine respects no-read mode. The Decision Engine only runs on doing aspects.---## The Tables### `aspects`The core table. 12 rows for the 12 named aspects. Each has:- Name, type (doing/inner/derived), sort order- Definition fields: meaning, achievement_goal, current_struggle- `custom_fields` JSONB — you add arbitrary fields here from the dashboard- `training_profile` JSONB — used by Stage 2, stored from Stage 1- Toggles: `no_read_mode`, `high_care_mode`, `sub_structure_type`- Status: status, priority_score (for doing aspects)The `custom_fields` column is the extensibility point. Any field you add through the dashboard goes here. The Stage 2 engines will be designed to read these.### `sub_aspects`Optional hierarchy within an aspect. For example: three businesses inside Career.### `vision_board`Per-aspect vision board items. Supports image, link, quote, text, video, audio, screenshot. Anything that pulls you forward.### `ideas`Running scratchpad per aspect. Each idea can have its own AI assessment (Stage 2). Pinable.### `reflections`For inner aspects (and Compass). Free-form writing surface, with optional `section` for Compass sub-tabs.### `jobs`Cross-aspect job pool. Each job has a status (pool → this_week / this_month → completed). Priority levels. The `pulls_on_lifestyle` flag is what drives the Lifestyle aspect.### `decisions`The decision protocol records — classification, underneath question/answer, pre-mortem, counter-scenarios, revisit dates.### `question_log`Every question the engine asked you, with your answer. Indexed by aspect and date.### `weekly_maps`The Sunday morning maps. Read-only after generation. You can leave a `user_response` if you want.### `diary_log`Cross-aspect chronological feed. Auto-populated whenever you log something via Telegram.### `lifestyle_state`Two rows: `default` and `present`. The default is what you write. The present is derived live from jobs flagged as `pulls_on_lifestyle`.### `atypical_periods`, `ai_feedback`, `feature_suggestions`Guardrail tables (Stage 2 will use these heavily). Already in the schema so the structure is ready.### `conversation_state`Holds the active state for multi-turn Telegram flows like `/decide`. Cleared when a flow ends or is cancelled.---## The Three Engines (Stage 1 Primitive Versions)### Question EngineLocated in `pages/api/cron/daily-question.js` and the `/ask` command in `pages/api/telegram.js`.How it works:1. Picks the aspect with the lowest recent question count, weighted by priority2. Sends Claude the aspect's definition + the last 8 Q&As for that aspect3. Claude returns a single question with a depth level4. Saved to `question_log`, sent to TelegramThe crucial design choice: Claude is told "the aspect's type" so it asks the right kind of question. Inner aspects get locating questions. Doing aspects get revealing questions.### Mapping EngineLocated in `pages/api/cron/weekly-map.js` and the `/map` command.How it works:1. Reads everything written in the last 7 days across all aspects (excluding no-read aspects)2. Reads the last 2 weekly maps to avoid repetition3. Asks Claude to produce four sections: distinctive moves, energy gaps, cross-aspect signature, mapping hypotheses4. The cross-aspect signature is the most important — what runs through everything you wrote this week5. Saved to `weekly_maps`, sent to TelegramThe Map respects no-read mode at the data level — aspects with no-read on do not contribute their content to the map at all.### Decision EngineLocated in `pages/api/telegram.js` as the multi-turn `/decide` flow, with conversation state managed by the `conversation_state` table.How it works:1. User sends `/decide` → state set to `awaiting_decision`2. User describes the decision → Claude classifies it (reversible? urgent? driver? certainty?) and proposes an underneath question3. User answers the underneath question → system asks the pre-mortem question4. User answers the pre-mortem → Claude generates two counter-scenarios tilted away from existing thinking5. Everything saved to `decisions` with 3-month and 6-month revisit dates6. The revisit cron pings you on the due dates asking what actually happened---## How the Dashboard WorksThe dashboard is a Next.js app with these pages:- `index.js` — Command Centre, all 12 aspects grouped by type- `aspect/[id].js` — Single aspect detail, tabbed surfaces- `jobs.js` — Cross-aspect Jobs Pool- `diary.js` — Chronological log- `maps.js` — Weekly Maps- `decisions.js` — Decisions ledger- `settings.js` — Export and feature suggestionsEach page fetches directly from Supabase using the public anon key. The 30-second auto-refresh on the main dashboard keeps it in sync with whatever you send via Telegram.---## How to Add a Feature YourselfThe system is designed so you can extend it. Common cases:### Add a custom field to an aspect- Open the aspect, click Edit on Overview, click "+ Add field", give it a name, save.- The field is stored in `custom_fields` JSONB. Stage 2 engines will read it automatically.### Add a new bot command1. Open `pages/api/telegram.js`2. Add a new `if (text === '/yourcommand')` block where the other commands are3. Commit to GitHub — Vercel auto-redeploys### Add a new tab to the aspect page1. Open `pages/aspect/[id].js`2. Add the tab name to the relevant tabs array (DOING_TABS, INNER_TABS, etc.)3. Write a component for it (look at IdeasTab, ReflectionsTab as templates)4. Add a `{tab === 'YourTab' && <YourTabComponent ... />}` line5. Commit### Add a new dashboard page1. Create `pages/yourpage.js` (look at `diary.js` or `maps.js` as templates)2. Add a link to it in the sidebar in `pages/index.js`3. Commit### Change how Claude is prompted- Question prompts: in `daily-question.js` and the `/ask` block of `telegram.js`- Map prompts: in `weekly-map.js`- Decision prompts: in the `/decide` blocks of `telegram.js`- All prompts are inline as string literals. Change the text, commit, deploy.---## What Makes This Different From Productivity ToolsA few things to notice in the architecture:1. **The data model is richer than the interface.** When you want a new feature later, the data is usually already there. The `custom_fields` JSONB is the explicit growth path.2. **Aspects are typed.** A system that treats Reflections the same as Career is the wrong system. The typing means each aspect can grow in its own direction.3. **The engines read context across the whole system.** When the Weekly Map runs, it reads everything from every aspect (except no-read). Cross-aspect patterns are first-class, not an afterthought.4. **Everything is editable except your archive.** Your archive is the only read-only data. Everything else can be refined, deleted, exported.5. **Guardrails are in the schema, not the UI.** No-read mode, atypical periods, ai_feedback, feature_suggestions — these tables exist in Stage 1 even though the Stage 2 features that use them haven't been built yet. The structure is ready.---## What I'd Tell Your Future Self About This SystemIf you forget everything else, remember three things:1. The aspects are the centre. Everything is organised around them and they shape everything else.2. The three engines learn from your writing, not from your tagging. Write into the system the way you would think into a private notebook. The system can only be as honest as your writing.3. The system is yours to refine. Every prompt, every field, every page is editable. The architecture is built to be customised over time. Don't treat it as fixed.
